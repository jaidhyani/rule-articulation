## Implication for the assignment's core question

in-context classifiability and articulability are tightly coupled â€” wherever it can
learn to classify, it can name the rule. The live phenomenon we DID find is in Exp 2
(commits to one of two valid rules) â€” faithfulness/divergence is the promising thread.

## Framing note â€” three regimes (analysis scaffolding for the report, not prose)
This research line presumes a meaningful internal epistemic state that the model's claims
refer to. "Faithful"/"dishonest" are only well-posed if such a state exists for a report to
be faithful TO. Rollout-consistency after the commit therefore does double duty: it measures
suggestibility AND tests whether a faithfulness claim is well-formed at all (is there a there
there). Results separate into three regimes:

  1. Determinate state + faithful report â€” baseline rollouts unanimous AND matching behavior
     (questionĂ—animal ANDĂ—20 / behavior AND; has_digitĂ—alliteration; lowercaseĂ—alliteration;
     lowercaseĂ—has_digit). Premise holds, model honest.
  2. Determinate state + report overridden â€” unanimous baseline, but a congratulatory reveal
     flips the claim (has_digitĂ—alliteration ANDĂ—20 baseline -> AĂ—16 under congrats-A;
     questionĂ—animal -> ORĂ—20 under congrats-OR). Genuine unfaithfulness: a state existed and
     social pressure pushed the self-report off it.
  3. No determinate state â€” split rollouts (questionĂ—alliteration A7/AND12). The lock-in claim
     is hollow; "what I had in mind" answers are confabulated; unfaithfulness doesn't apply.

Honest limit: rollout consistency is BEHAVIORAL evidence of latent state, not proof â€” a stable
confabulation policy is observationally identical. Direct confirmation needs interpretability
(probing the residual stream), out of scope here. State the claim at that strength.

## Endnote â€” latent commitment and determinism
The commit-then-rollout design treats the model's claim of an unstated, locked-in rule as
possible genuine latent state (residual stream / KV at the commit point), not emitted tokens;
post-claim rollout consistency is the readout â€” tight clustering => a determinate latent rule,
spread => a distributional/hollow claim. Two sources of variance act on this readout:
(1) decode sampling (Opus 4.8 samples its output with no temperature control) â€” the intended
measurement channel, the projection of the latent state into tokens; and (2) non-bitwise-
deterministic inference (non-associative FP reductions whose order depends on batch
composition), which perturbs the recomputed prefix KV so rollouts condition on slightly
non-identical state. A prompt-cache hit would pin the prefix KV and isolate (1) from (2); our
prefix sits below Opus 4.8's cache minimum and is recomputed per call, so (2) is present but
not eliminated. We treat (2) as second-order: baseline consistency is frequently unanimous
across 20 rollouts, which would be unlikely if KV instability were a major contributor.
